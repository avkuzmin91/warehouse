# # СТРАНИЦА СОЗДАНИЯ ТОВАРА

---

# 1. Цель
Реализовать страницу создания нового товара справочника.

Страница предназначена для:

- создания нового товара
- валидации введённых данных
- сохранения товара в системе
- отправки данных в API

---

# 2. Зависимости

Использовать Data Model: Product

---

# 3. Доступ

- доступ только для роли **admin**
- при отсутствии доступа отображение 403 страницы

---

# 4. Маршрут

/dictionaries/products/new

---

# 5. UI структура страницы

## 5.1 Общая структура

Страница должна содержать:

- Хлебные крошки
- Заголовок
- форму
- Кнопки управления

---

## 5.2 Хлебные крошки

Формат:
Главная / Справочники / Справочник товаров / Создание товара

---

## 5.3 Заголовок

Текст: "Создание товара"

---

## 5.4 Форма

### Общие правила

Форма отображается в виде вертикального списка полей.

- поля соответствуют Data Model: Product
- обязательные поля помечены "*"
- форма вертикальная

---

### Поля формы

1. Название товара
   - тип: текстовое поле
   - обязательное
   - отображать "*" (красная звёздочка)

2. Тип товара

   - Тип: select

   - Значения: выбор из доступных по модели данных

   - обязательное
     

3. Артикул товара

   - тип: текстовое поле
   - обязательное
   - уникальное

4. Поставщик
   - тип: текстовое поле
   - необязательное

5. Фотография (image_url)
   - тип: file upload
   - форматы: jpg, png, heic
   - после выбора отображается превью

6. Не актуален
   - тип: checkbox
   - по умолчанию: false

---

## 6. Поведение

## 6.1 Инициализация

- Страница для создания товара инициируется по кнопке "Создать", нажатой на странице dictionaries/products
- форма открывается пустой
- is_active = false

---

## 6.2 Валидация

При нажатии "Сохранить":

Проверяется:

- name заполнен
- type выбран
- sku заполнен

---

### Ошибки:

- показать общее сообщение: "Заполните обязательные поля"

- подсветить конкретные поля

---

## 6.3 Кнопка "Сохранить"

- текст: "Сохранить"
- расположение: под полями, справа
- имеет hover-эффект
- поведение, если валидация успешна:
  - выполнить POST /products
  - → переход на /dictionaries/products
  
  \### Request:
  
  \```json
  {
    "name": "string",
    "type": "clothes | electronics",
    "sku": "string",
    "supplier": "string",
    "is_active": false
  }
  
  Файл отправляется отдельно (multipart/form-data)
  
  ### Ошибка API
  
  Если backend вернул ошибку:
  
  Примеры:
  
  - "SKU уже существует"
  
  Отобразить сообщение пользователю

## 6.4 Кнопка "Отмена"

- текст: "Отмена"
- расположение: под полями, справа от кнопки "Сохранить"
- имеет hover-эффект
- поведение:
    - Товар не сохраняется
    - → переход на /dictionaries/products

# 7. UI требования

- форма расположена по центру экрана
- ограничение ширины (max-width)
- адаптивность (mobile-friendly)
- отступы между полями

---

# 8. Ограничения

- не использовать модальные окна

---

# 9. Безопасность

- доступ к API только для admin
- backend проверяет роль пользователя
- frontend не является источником прав доступа
- backend валидирует: уникальность SKU

---

# 10. Проверка

1. ## Сценарий 1: Успешное создание

   1. Войти под admin
   2. Открыть /dictionaries/products
   3. Нажать "Создать"
   4. Заполнить форму
   5. Нажать "Сохранить"

   Ожидается:

   - редирект на список
   - товар появился в таблице

   ------

   ## Сценарий 2: Пустые поля

   1. Нажать "Сохранить" без заполнения

   Ожидается:

   - сообщение об ошибке
   - подсветка полей

   ------

   ## Сценарий 3: Дубликат SKU

   1. Ввести существующий SKU
   2. Нажать "Сохранить"

   Ожидается:

   - ошибка "SKU уже существует"

   ------

   ## Сценарий 4: Отмена

   1. Нажать "Отмена"

   Ожидается:

   - переход на список
   - данные не сохраняются

   ------

   ## Проверка прав

   1. зайти под user или manager
   2. открыть /dictionaries/products/new

   Ожидается:

   - 403
