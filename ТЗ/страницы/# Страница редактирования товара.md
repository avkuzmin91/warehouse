# # СТРАНИЦА РЕДАКТИРОВАНИЯ ТОВАРА

---

# 1. Цель

Реализовать страницу **редактирования** существующего товара в справочнике (обновление полей и фото).

Страница предназначена для:

- просмотра и изменения атрибутов товара, загруженных с сервера
- валидации введённых данных
- сохранения изменений в системе
- отправки данных в API (частичное обновление)

---

# 2. Зависимости

- Data Model: **Product** (те же поля, что и при создании)
- Ссылка на согласование полей, хлебных крошек и отсутствия отдельного h1: ТЗ `02 Глобальные компоненты / # Хлебные крошки.md`
- Сопоставление с ТЗ: `# Страница создания товаров.md` (состав полей, UX формы) — **реализация редактирования опирается на тот же набор полей**, с отличием в инициализации (данные с сервера) и HTTP-методе сохранения

---

# 3. Доступ

- доступ только для роли **admin**
- при отсутствии доступа — отображение **403** (см. общий подход в приложении)

---

# 4. Маршрут

- `/dictionaries/products/{id}/edit`
- `id` — идентификатор существующего товара (тот же, что в `GET /products/{id}`)

**Точка входа (рекомендуемая):** переход с **карточки/просмотра товара** (кнопка «Редактировать») и/или из контекстного сценария; прямой переход по URL — допустим, при валидном `id`

---

# 5. UI структура страницы

## 5.1 Общая структура

Страница должна содержать (сверху вниз внутри стеклянной карточки):

- **Хлебные крошки** (см. п. 5.2) — **название текущего шага** в последних сегментах цепочки; отдельный заголовок **(h1)** **не** выводится
- **Форма** (редактируемые поля — как в п. 5.3; **дополнительно** — только просмотр дат в п. 5.4)
- **Кнопки управления** («Сохранить», «Отмена») — согласовать по расположению с ТЗ создания

**Состояния загрузки/ошибки:** до появления данных товара допускается индикатор загрузки; при `404` или ошибке API — сообщение и отсутствие формы (или сценарий возврата к списку)

---

## 5.2 Хлебные крошки

- Реализуются **общим компонентом** `Breadcrumbs` (см. глобальное ТЗ о крошках)
- **Размещение:** первый блок внутри `auth-card`, над формой
- **URL:** `pathname` включает сегменты `…/products/…/edit` — **маппинг** динамического `id` и `edit` задаётся в `utils/breadcrumbLabels.ts` (например, `id` → **«Карточка»** или уточнённый ярлык по продукту, `edit` → **«Редактирование»**; при развитии требований — единая политика с глобальным ТЗ)

**Ориентировочная цепочка (пример):**

`Главная / Справочники / Товары / … / Редактирование`

(конкретный вид средних сегментов зависит от принятого маппинга; последний **не** кликабелен, текущий экран)

---

## 5.3 Форма

### Общие правила

- Вертикальный список полей, **те же сущностные поля**, что в ТЗ **«# Страница создания товаров.md»** (п. 5.3: название, тип, артикул, поставщик, фото, флаг актуален / «не актуален»)
- Подписи обязательных полей с пометкой `*`
- При открытии страницы поля **предзаполняются** из ответа **GET** `/products/{id}` (включая отображение текущего фото, если `image_url` задан)
- **SKU:** при смене значения по-прежнему действует правило уникальности (конфликт — как при создании, ответ API)

### Отличия от «создания»

- **Инициализация:** не пустая форма, а данные с **GET** `/products/{id}`
- **Сохранение:** **PATCH** `/products/{id}` (см. п. 6), `multipart/form-data` при передаче файла и/или полей как в существующем **backend** (`Form`), частичные поля — по контракту API (см. п. 6.3)
- **Метаданные:** на экране **отображаются** даты создания и последнего изменения (см. п. 5.4), **не** входят в тело **PATCH** и **не** редактируются пользователем

---

## 5.4 Блок дат (только просмотр)

- **Поля (всегда только чтение):**
  1. **Дата создания** — из поля `created_at` (или аналога в **Product** / ответе **GET** `/products/{id}`)
  2. **Дата последнего изменения** — из поля `updated_at` (если `null` / отсутствует — отобразить **«—»** или **«Нет»** по согласованной подписи)
- **Визуально:** не интерактивные элементы (текст, `read-only` поля **без** фокус-редактирования, или отдельный нейтральный блок) — **без** `input` для ввода дат
- **Поведение:** значения **обновляются** после **GET** при открытии; после **успешного PATCH** при необходимости **перезапросить** `GET` `/products/{id}` **или** обновить **только** `updated_at` из ответа, если договорён **полный** объект в ответе; минимум — **актуальные** даты **после следующего** захода на экран/обновлении
- **Не отправлять** `created_at` / `updated_at` в **PATCH** — даты **выставляет сервер** при создании/обновлении

---

# 6. Поведение

## 6.1 Инициализация

- При заходе на маршрут: запрос **GET** `/products/{id}`
- При успехе: заполнить форму значениями; превью изображения — по URL, возвращаемому API; **отобразить** **даты создания и последнего изменения** (п. 5.4) из **GET** — **не** как редактируемые поля
- При **404** или ошибке: показать сообщение (например «Товар не найден») и/или **редирект** на `/dictionaries/products` — по согласованной UX-политике
- `id` в пути **не** редактируется в форме (косвенно задаёт маршрут)

---

## 6.2 Валидация

Аналогично странице **создания** (обязательность `name`, `type`, `sku` перед сохранением, общее сообщение **«Заполните обязательные поля»**, подсветка полей) — **если** при редактировании снова требуется полное заполнение перед **PATCH**; иначе — валидация **отправляемого** подмножества по правилам продукта/бэкенда

---

## 6.3 Кнопка «Сохранить»

- **Текст:** «Сохранить» (при необходимости: «Сохранить изменения»)
- **Расположение / hover:** согласовать с ТЗ создания
- При успешной валидации: **PATCH** `/products/{id}` (тело: `FormData` / поля, как в реализации `update_product` в backend: `name`, `type`, `sku`, `supplier`, `is_active`, `image` — **не** передавать `null` полями, снимаемые с формы, если бэкенд требует явной семантики «пусто / не обновлять»)
- **Успех:** переход на **список** товаров `/dictionaries/products` (как в ТЗ создания) **или** на **карточку** товара `…/dictionaries/products/{id}` — зафиксировать один сценарий в реализации; по умолчанию — **список** для согласованности с созданием
- **Ошибки API:**  
  - «**SKU уже существует**» (конфликт уникальности при смене артикула)  
  - **404** (товар удалён)  
  - **400** «**Нет данных для обновления**» (если ни одно поле не изменено и не послано) — обработка по соглашению с бэкендом

**Пример контракта (ориентир — текущий backend):**

- Метод: `PATCH`  
- `Content-Type: multipart/form-data` при наличии файла  
- Сообщение об успехе: по полю `message` ответа (строка вроде «**Обновлено**»)

---

## 6.4 Кнопка «Отмена»

- **Поведение:** изменения **на сервер** не отправлять
- **Переход:** на **список** `/dictionaries/products` (как в ТЗ создания) или **назад** (история) — в реализации зафиксировать один вариант; **рекомендуемый** — **список** для согласованности

---

# 7. UI требования

- Как в ТЗ создания: **центрирование** блока, **max-width**, **адаптив**, отступы между полями
- Текущее изображение: **превью**; замена — через `file` input; при смене файла — **новый** `PATCH` с полем `image` (по правилам backend)
- **Блок дат** (п. 5.4): визуально **отделён** от редактируемых полей (например, группа **«Служебная информация»** / подзаголок) или **после** редактируемой формы, но **без** возможности изменения

---

# 8. Ограничения

- Без **модальных** окон для основной формы (как в ТЗ создания)
- **Inline-редактирование** в списке товаров — вне scope данного ТЗ (при появлении — отдельное соглашение)

---

# 9. Безопасность

- **GET/PATCH/DELETE** по товарам — **только** для **admin** (как в backend `Depends(get_current_admin)`)
- **Frontend** не должен рассчитывать на скрытие: проверка роли на **сервере** обязательна
- **Идентификатор** в пути: право «редактировать **этот** товар» определяется **бэкендом**

---

# 10. Проверка (сценарии)

1. **Успешное открытие**  
   Под **admin** открыть `/dictionaries/products/{валидный id}/edit` — форма **заполнена**, крошки **отображаются**; **видны** **дата создания** и **дата последнего изменения** (только чтение)

2. **Сохранение изменения названия (и др. поля)**  
   Изменить поле → **Сохранить** — **успех**, товар **обновлён**; **дата последнего изменения** (после обновления данных) **отражает** момент правки; **дата создания** **без** изменения

3. **Нередактируемость дат**  
   Убедиться, что **нельзя** ввести или изменить даты вручную; при отсутствии `updated_at` (если в модели) — **корректная** отметка (например **«—»**)

4. **Смена фото**  
   Загрузить новый файл → **Сохранить** — превью/список отражают **новую** ссылку на фото (по правилу backend)

5. **Валидация**  
   Очистить обязательное поле (если полития не допускает пустого **PATCH** для этого поля) — сообщение, подсветка

6. **Дублирование SKU**  
   Поставить **SKU**, уже существующий у **другого** товара — сообщение **«SKU уже существует**» (или иной текст, совпадающий с **API**)

7. **Отмена**  
   **Отмена** — возврат **без** смены данных

8. **Нет доступа**  
   Под **user/manager** открыть **edit-URL** — **403** (и/или **AccessDenied**)

9. **Некорректный id**  
   Несуществующий `id` — **404** и корректное **сообщение/редирект** по п. 6.1

---

## Связь с существующим API (на момент составления ТЗ)

- **GET** `/products/{item_id}` — тело **Product** (используется для **prefill**), в т.ч. `created_at`, `updated_at` **только** для **отображения** в п. 5.4
- **PATCH** `/products/{item_id}` — **Form**-поля (и **File** `image`), **частичное** обновление, ответ `{ message: string }` — **без** ввода дат с клиента

ТЗ можно уточнять после **реализации** маршрута в `App.tsx` (имя **route**, слой **страницы/компонента**).
