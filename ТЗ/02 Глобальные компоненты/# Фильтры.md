# COMPONENT: FILTERS PANEL

---

# 1. Цель

Реализовать переиспользуемый компонент панели фильтров.

Компонент предназначен для:

- фильтрации списков данных
- управления параметрами запроса к API
- синхронизации фильтров с URL (query params)

---

# 2. Область применения

Используется на страницах:

- /dictionaries/products
- /dictionaries/clients
- другие списочные страницы

---

# 3. Общая структура

Компонент представляет собой горизонтальную панель над таблицей.

Содержит:

- поля фильтрации
- кнопку сброса

---

# 4. UI структура

---

## 4.1 Расположение

- располагается над таблицей
- внутри основной карточки страницы
- под хлебными крошками и кнопкой "Создать"

---

## 4.2 Визуальное оформление

- горизонтальный layout
- элементы выровнены по одной линии
- отступы между элементами: 12–16px
- перенос на новую строку при нехватке ширины (responsive)

---

## 4.3 Состав элементов (универсальный)

---

### 1. Поле поиска (search)

- тип: текстовое поле
- placeholder: "Поиск..."
- фильтрует по основному текстовому полю (например, name)

---

### 2. Select фильтры

Примеры:

#### Тип

- тип: select
- значения:
  - (зависит от сущности)

---

#### Статус (is_active)

- тип: select
- значения:
  - Да
  - Нет

---

### 3. Кнопка "Сбросить"

- текст: "Сбросить"
- очищает все фильтры
- возвращает значения по умолчанию

---

# 5. Поведение

---

## 5.1 Общая логика

При изменении любого фильтра:

- обновляется состояние фильтров
- выполняется запрос к API
- обновляется таблица

---

## 5.2 Debounce (для поиска)

- задержка: 300–500 мс
- запрос выполняется после остановки ввода

---

## 5.3 Комбинирование фильтров

- все фильтры применяются одновременно
- формируется единый набор параметров

---

## 5.4 Сброс фильтров

При нажатии "Сбросить":

- очищаются все поля
- сбрасываются query params
- выполняется запрос без фильтров

---

# 6. Состояние (State)

---

## 6.1 Внутреннее состояние

Пример:

```ts
{
  search: string,
  type: string | null,
  is_active: boolean | null
}
```

---

## 6.2 Источник истины

- состояние хранится в React state
- синхронизируется с URL

------

# 7. Работа с URL (Query Params)

------

## 7.1 Синхронизация

Фильтры должны сохраняться в URL.

------

## 7.2 Пример URL

/dictionaries/products?search=shirt&type=clothes&is_active=true&page=1

## 7.3 Правила

- при изменении фильтра → обновляется URL
- при открытии страницы → фильтры берутся из URL

------

# 8. API интеграция

------

## 8.1 Формирование запроса

Фильтры передаются в GET запрос: GET /products?search=shirt&type=clothes&is_active=true&page=1&limit=20

## 8.2 Общие правила

- не передавать пустые значения
- использовать только активные фильтры

------

# 9. Переиспользование

------

## 9.1 Конфигурация

Компонент должен принимать конфигурацию:

```typescript
filtersConfig = [
  {
    name: "search",
    type: "text",
    placeholder: "Поиск по названию"
  },
  {
    name: "type",
    type: "select",
    options: ["clothes", "electronics"]
  },
  {
    name: "is_active",
    type: "select",
    options: [true, false]
  }
]
```

## 9.2 Поведение

- компонент не знает бизнес-логики
- только отображает фильтры и возвращает значения

------

# 10. Ограничения

- не использовать модальные окна
- не делать inline фильтрацию без API
- не хранить состояние только в UI (обязательно URL)

------

# 11. Безопасность

- фильтры не влияют на права доступа
- backend валидирует параметры

------

# 12. Сценарии тестирования

------

## Сценарий 1: Поиск

1. Ввести текст в поле поиска

Ожидается:

- через debounce выполняется запрос
- таблица обновляется

------

## Сценарий 2: Выбор фильтра

1. Выбрать значение в select

Ожидается:

- выполняется GET запрос
- данные обновляются

------

## Сценарий 3: Комбинация фильтров

1. Ввести поиск
2. Выбрать тип

Ожидается:

- применяются оба фильтра

------

## Сценарий 4: Сброс

1. Нажать "Сбросить" (иконка)

Ожидается:

- фильтры очищены
- URL очищен
- таблица обновлена

------

## Сценарий 5: Обновление страницы

1. Открыть URL с параметрами

Ожидается:

- фильтры применяются автоматически
- значения отображаются в UI

------

## Сценарий 6: Пустые параметры

1. Очистить фильтры

Ожидается:

- параметры не передаются в API
