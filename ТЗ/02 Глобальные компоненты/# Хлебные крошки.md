# ГЛОБАЛЬНЫЙ КОМПОНЕНТ: ХЛЕБНЫЕ КРОШКИ (BREADCRUMBS)

---

# 1. Цель

Реализовать глобальный компонент хлебных крошек для навигации по приложению.

Компонент предназначен для:
- отображения текущего пути пользователя
- упрощения навигации между уровнями страниц
- стандартизации UI навигации

---

# 2. Область применения

Breadcrumbs отображаются на внутренних страницах приложения, в том числе:

- **/home** — одна крошка «Главная» (текущая страница, без ссылок выше)
- /dictionaries и вложенные маршруты
- /inventory
- /dictionaries/products/* и прочие разделы
- /users
- другие защищённые страницы
- экраны «Проверка доступа…» (ожидание проверки роли) и страница 403 (при отображении в рамках защищённого layout)

---

# 3. Не отображается

Компонент НЕ отображается на страницах:

- /auth
- /auth/register

---

# 4. Структура компонента

Breadcrumbs представляет собой цепочку навигационных элементов:

Пример (между сегментами — пробелы вокруг разделителя «/»):

`Главная / Справочники / Товары / Создание товара`

(визуально: неразрывный пробел, «/», пробел, затем следующий сегмент)

---

# 5. Поведение

## 5.1 Формирование

- путь формируется автоматически на основе текущего URL (`useLocation` → `pathname`)
- каждый сегмент URL преобразуется в читаемое название (см. п. 6)
- логика маппинга вынесена в отдельный модуль (например, `utils/breadcrumbLabels.ts`)

---

## 5.2 Навигация

- все элементы, кроме последнего, кликабельны
- последний элемент (текущая страница) не кликабелен, с атрибутом `aria-current="page"`

---

## 5.3 Пример

URL: /dictionaries/products/new

Отображение: Главная / Справочники / Товары / Создание товара

---

# 6. Правила отображения названий

## 6.1 Статические маппинги (фрагменты пути)

- `home` → Главная
- `dictionaries` → Справочники
- `products` → Товары
- `users` → Пользователи
- `inventory` → Учет товаров
- сегменты разделов справочника: `clients` → Клиенты, `colors` → Цвета, `sizes` → Размеры и т.д.

---

## 6.2 Создание и формы

- сегмент `new` после известного раздела справочника → осмысленный заголовок, например «Создание товара», «Создание клиента» (зависит от родительского сегмента)
- сегмент `new` без подходящего контекста → **Создание**

---

## 6.3 Динамические сегменты

- идентификатор карточки (в пути `/dictionaries/{section}/{id}`) → **Карточка**
- сегмент `edit` → **Редактирование**

---

# 7. UI требования

## 7.1 Размещение

- хлебные крошки отображаются **в верхней части основного контента**
- **внутри стеклянной карточки** (класс `auth-card` / тот же визуальный контейнер, что и формы, таблицы, списки)
- **первый дочерний блок** внутри `<section class="auth-card …">` (сразу под «стеклом»), **не** вынесены в отдельную полосу над карточкой и **не** в общем `AppLayout` над `<Outlet />`
- на внутренних страницах **отдельный заголовок уровня h1** под крошками **не** используется: наименование текущего раздела — в **последней крошке** (исключения: экраны `/auth` и `/auth/register` без крошек, со своим заголовком)

## 7.2 Внешний вид

- горизонтальное расположение, перенос строки при нехватке ширины (flex, wrap)
- разделитель между сегментами: символ «/» с **пробелами с обеих сторон** (в т.ч. визуально устойчивые к схлопыванию пробелов в вёрстке, например `white-space: pre` у разделителя)
- **цвета** (согласованы с ранее использовавшимися `dict-breadcrumbs`):
  - обводка цепочки: нейтральный приглушённый текст
  - кликабельные сегменты: акцентный (primary-soft)
  - текущий (последний) сегмент: основной мягкий текст (`text-soft`), начертание **600**
- кликабельные сегменты: `font-weight: 600`, подчёркивание при **hover**
- снизу у блока крошек: отступ до следующего элемента карточки (например, **20px**), чтобы визуально отделить путь от формы, таблицы, списка ссылок и т.д.

---

# 8. Техническая реализация (Frontend)

## 8.1 Компонент

- общий React-компонент **`Breadcrumbs`** (например, `src/components/Breadcrumbs.tsx`)
- формирование цепочки: **`buildBreadcrumbsFromPathname(pathname)`** в `src/utils/breadcrumbLabels.ts` (или аналог)

## 8.2 Подключение

- компонент **подключается на уровне страниц** (и при необходимости в вложенных маршрутах с той же карточкой), **первой строкой внутри** `auth-card`, а **не** единожды в `AppLayout` вместе с `Header` и `Outlet`
- `AppLayout` содержит только шапку и `Outlet` для контента

## 8.3 Пример вёрстки страницы

**Главная** (`/home`):

```jsx
<main className="page page--center">
  <section className="auth-card home-card">
    <Breadcrumbs />
    { /* далее контент: список разделов и т.д. — без h1, название в последней крошке */ }
  </section>
</main>
```

Другая внутренняя страница (тот же принцип: крошки первыми в `auth-card`):

```jsx
<main className="page page--center">
  <section className="auth-card dict-hub-card">
    <Breadcrumbs />
    { /* контент: таблица, кнопки, форма — без дублирующего h1 */ }
  </section>
</main>
```

## 8.4 Стили

- CSS-классы: `breadcrumbs`, `breadcrumbs__list`, `breadcrumbs__item`, `breadcrumbs__link`, `breadcrumbs__current`, `breadcrumbs__sep` (см. `src/style.css`)

---

# 9. Безопасность и источник данных

- крошки — только отображение; кликабельные ссылки ведут на маршруты приложения; **фактические права** по-прежнему проверяет backend; при 403 — отдельная страница/состояние, при этом путь в URL и крошки могут отражать запрошенный маршрут
