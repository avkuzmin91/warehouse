# COMPONENT: SORTING

---

# 1. Цель

Реализовать универсальный механизм сортировки данных в таблицах.

Компонент предназначен для:

- изменения порядка отображения данных
- управления параметрами сортировки
- интеграции с Query State и API

---

# 2. Область применения

Используется:

- в таблицах (Table Component)
- на страницах списков (products, clients и др.)

---

# 3. Общая концепция

Сортировка осуществляется через клики по заголовкам колонок таблицы.

---

# 4. UI поведение

---

## 4.1 Триггер сортировки

Сортировка вызывается:

- кликом по заголовку колонки

---

## 4.2 Состояния сортировки

Каждая колонка может иметь 3 состояния:

1. Нет сортировки
2. По возрастанию (asc)
3. По убыванию (desc)

---

## 4.3 Переключение

При каждом клике:

```text
нет → asc → desc → нет
```

## 4.4 Визуализация

- отображается иконка:
  - ↑ для asc
  - ↓ для desc
- активная колонка визуально выделяется

------

# 5. Поведение

------

## 5.1 Изменение сортировки

При клике:

- обновляется sort в Query State
- обновляется URL
- выполняется API запрос

------

## 5.2 Сброс сортировки

Если состояние возвращается в "нет":

- параметр sort удаляется из URL
- выполняется запрос без сортировки

------

## 5.3 Одна активная сортировка

- допускается сортировка только по одной колонке

------

# 6. Интеграция с Query State

------

## 6.1 Формат

```typescript
sort: {
  field: string,
  direction: "asc" | "desc"
}
```

## 6.2 Пример

/dictionaries/products?sort=created_at_desc

---

## 6.3 Формирование параметра

sort = {field}_{direction}

Примеры:

- name_asc
- created_at_desc

# 7. API интеграция

------

## 7.1 Запрос

GET /products?sort=name_asc

---

## 7.2 Требования к backend

- валидировать поле сортировки
- валидировать направление
- игнорировать некорректные значения

------

# 8. Ограничения

- не использовать клиентскую сортировку
- не допускать множественную сортировку (multi-sort)
- не хранить состояние только в UI

------

# 9. Безопасность

- backend проверяет допустимые поля
- frontend не доверяется

------

# 10. Сценарии тестирования

------

## Сценарий 1: Сортировка по возрастанию

1. Нажать на колонку "Название"

Ожидается:

- sort=name_asc
- данные отсортированы

------

## Сценарий 2: Сортировка по убыванию

1. Нажать повторно

Ожидается:

- sort=name_desc

------

## Сценарий 3: Сброс

1. Нажать третий раз

Ожидается:

- сортировка удалена
- данные без сортировки

------

## Сценарий 4: Обновление страницы

1. Открыть URL с sort

Ожидается:

- сортировка применена

------

## Сценарий 5: Переключение колонок

1. Отсортировать по name
2. Затем по created_at

Ожидается:

- сортировка по name сбрасывается
- активна только created_at

------

## Сценарий 6: Проверка API

1. Выполнить сортировку

Ожидается:

- запрос содержит параметр sort
