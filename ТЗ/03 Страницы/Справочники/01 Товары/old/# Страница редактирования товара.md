# СТРАНИЦА: РЕДАКТИРОВАНИЕ ТОВАРА

---

# 1. Цель

Реализовать страницу редактирования товара справочника.

Страница предназначена для:
- редактирования данных
- сохранения изменений в системе
- отправки обновлённых данных в API

---

# 2. Зависимости

Использовать Data Model:
- Product

---

# 3. Доступ

- доступ только для роли **admin**
- при отсутствии доступа:
  - отображение 403 страницы

---

# 4. Маршрут

- /dictionaries/products/{id}

---

# 5. UI структура страницы

---

## 5.1 Общая структура

Страница должна содержать:

- хлебные крошки
- форму редактирования
- кнопки управления

> ВАЖНО: отдельный заголовок (h1) НЕ используется, так как название страницы содержится в хлебных крошках.

---

## 5.2 Хлебные крошки

- реализуются через глобальный компонент Breadcrumbs
- формируются автоматически из URL

Для маршрута: /dictionaries/products/{id}

Отображение: Главная / Справочники / Товары / Редактирование товара

---

# 6. Форма

## 6.1 Общие правила

- поля соответствуют Data Model: Product
- форма заполняется данными, полученными из API
- обязательные поля помечены "*"

---

## 6.2 Загрузка данных

При открытии страницы:

- выполняется запрос: GET /products/{id}

- результат заполняет форму

---

## 6.3 Поля формы

### Название товара (name)

- тип: текстовое поле
- обязательное

---

### Тип товара (type)
- select
- значения:
  - clothes → "Одежда"
  - electronics → "Техника"
- обязательное

---

### Артикул товара (sku)
- текстовое поле
- обязательное
- уникальное (валидируется backend)

---

### Поставщик (supplier)
- текстовое поле
- необязательное

---

### Фотография (image_url)
- file upload
- форматы: jpg, png, heic
- при наличии существующего изображения — показывается превью

---

### Актуален (is_active)
- checkbox
- значение берётся из backend

---



# 7. Поведение

---

## 7.1 Инициализация страницы

---

### Способ открытия страницы

Страница редактирования товара открывается:

1. По клику на строку в списке товаров

На странице:
- /dictionaries/products

При нажатии на любую строку таблицы товара:
→ происходит переход на страницу редактирования

---

2. Маршрут перехода

/dictionaries/products/{id}

где:
- {id} — уникальный идентификатор товара

---

### Передача данных

После перехода:

- выполняется запрос: GET /products/{id}

- данные подгружаются в форму

---

### Поведение до загрузки данных

Пока данные загружаются:

- отображается loader (спиннер)
- форма неактивна (disabled state)

---

### Ошибки загрузки

Если товар не найден:

- показать страницу состояния:
  "Товар не найден"



## 7.1 Инициализация

- при загрузке страницы подтягиваются данные товара
- форма заполняется автоматически

---

## 7.2 Валидация

При нажатии "Сохранить":

Проверяется:

- name не пустой
- type выбран
- sku не пустой

---

### Ошибки:

- сообщение: "Заполните обязательные поля"
- подсветка полей

---

## 7.3 Сохранение

Если валидация успешна:

- выполняется запрос: PATCH /products/{id}

---

### Request:

```json
{
  "name": "string",
  "type": "clothes | electronics",
  "sku": "string",
  "supplier": "string",
  "is_active": false
}
```

- файл отправляется отдельно (multipart/form-data)

## 7.4 Успешное сохранение

- данные по товару обновляются
- редирект на: /dictionaries/products

## 7.5 Ошибка API

Примеры:

- "SKU уже существует"

Поведение:

- показать сообщение пользователю
- не сбрасывать форму

# 8. Кнопки

------

## 8.1 Сохранить

- текст: "Сохранить"
- справа под формой

Поведение:

- запускает валидацию
- отправляет PATCH запрос

------

## 8.2 Отмена

- текст: "Отмена"
- справа от "Сохранить"

Поведение:

- переход на /dictionaries/products
- изменения не сохраняются

------

# 9. UI требования

- форма по центру
- max-width
- адаптивность
- отступы между полями
- preview изображения (если есть)

------

# 10. Ограничения

- не использовать модальные окна
- не дублировать данные вне формы

------

# 11. Безопасность

- доступ только admin
- backend проверяет:
  - роль
  - существование товара
  - уникальность SKU

------

# 12. Служебные поля (Audit fields)

---

## 12.1 Общие правила

Служебные поля:
- отображаются только в режиме просмотра/редактирования
- НЕ редактируются пользователем
- заполняются автоматически backend

---

## 12.2 Поля

### Дата создания (created_at)
- тип: datetime
- источник: backend
- отображается в read-only виде

---

### Создал (created_by)
- тип: string (или user reference)
- отображает:
  - имя пользователя или email
- источник: backend

---

### Изменил (updated_by)
- тип: string (или user reference)
- отображает:
  - последний пользователь, изменивший товар
- источник: backend

---

### Дата изменения (updated_at)
- тип: datetime
- источник: backend

---

## 12.3 UI размещение

Служебные поля отображаются:

- внизу формы
- отдельным блоком "Информация о записи"
- визуально отделены от редактируемых полей (divider / subtle card)

---

## 12.4 Пример отображения

## Информация о записи

Создано: 12.01.2026 10:45
Создал: admin@system.com

Последнее изменение: 15.01.2026 14:20
Изменил: manager@system.com

------

# 13. Проверка (Сценарии тестирования)

---

## Сценарий 1: Открытие страницы редактирования

### Шаги:

1. Войти под пользователем с ролью admin
2. Перейти на страницу:
   /dictionaries/products
3. В таблице товаров нажать на любую строку товара

### Ожидаемый результат:

- происходит переход на:
  /dictionaries/products/{id}
- отображается loader во время загрузки
- после загрузки отображается форма с данными товара

---

## Сценарий 2: Успешное редактирование товара

### Шаги:

1. Открыть существующий товар
2. Изменить поле "Название товара"
3. Нажать "Сохранить"

### Ожидаемый результат:

- выполняется запрос:
  PATCH /products/{id}
- данные обновляются в системе
- происходит редирект на:
  /dictionaries/products
- в списке отображаются обновлённые данные

---

## Сценарий 3: Валидация обязательных полей

### Шаги:

1. Открыть товар
2. Очистить обязательное поле "Название товара"
3. Нажать "Сохранить"

### Ожидаемый результат:

- отображается сообщение:
  "Заполните обязательные поля"
- поля с ошибками подсвечены
- запрос в API НЕ отправляется

---

## Сценарий 4: Ошибка уникальности SKU

### Шаги:

1. Открыть товар
2. Изменить SKU на уже существующий
3. Нажать "Сохранить"

### Ожидаемый результат:

- backend возвращает ошибку:
  "SKU уже существует"
- ошибка отображается пользователю
- форма остаётся открытой, данные не сбрасываются

---

## Сценарий 5: Отмена редактирования

### Шаги:

1. Открыть товар
2. Изменить любое поле
3. Нажать "Отмена"

### Ожидаемый результат:

- пользователь перенаправлен на:
  /dictionaries/products
- изменения НЕ сохраняются

---

## Сценарий 6: Проверка загрузки данных

### Шаги:

1. Открыть страницу:
   /dictionaries/products/{id}

### Ожидаемый результат:

- данные заполняются из API:
  GET /products/{id}
- отображаются корректные значения:
  - name
  - type
  - sku
  - supplier
  - image (preview)
  - is_active

---

## Сценарий 7: Проверка audit-полей

### Шаги:

1. Открыть товар

### Ожидаемый результат:

Отображаются только для просмотра:

- Дата создания
- Кто создал
- Дата последнего изменения
- Кто изменил

И они:

- не редактируются
- совпадают с backend данными

---

## Сценарий 8: Доступ без прав

### Шаги:

1. Войти под user или manager
2. Открыть:
   /dictionaries/products/{id}

### Ожидаемый результат:

- доступ запрещён
- отображается 403 страница
  или происходит редирект на /auth

---

## Сценарий 9: Несуществующий товар

### Шаги:

1. Перейти по адресу:
   /dictionaries/products/invalid-id

### Ожидаемый результат:

- отображается сообщение:
  "Товар не найден"
- кнопка "Назад" ведёт на:
  /dictionaries/products

---

## Сценарий 10: Проверка UI блокировки загрузки

### Шаги:

1. Открыть страницу редактирования

### Ожидаемый результат:

- пока идет загрузка:
  - форма disabled
  - отображается loader
- после загрузки форма активна
